Day 1  ·  10:30 – 11:00

The clinical data landscape — and why it is fragmented

Health data sits in silos: each uses its own schema, its own codes, its own conventions. The Semantic Web provides the common language that lets these silos speak to each other without merging or copying data.

OMOP CDM
Observational health data model. Standardised tables for person, condition_occurrence, drug_exposure. OHDSI ecosystem: Athena, Atlas, Achilles.
FHIR / HL7
HL7 FHIR — resource-based REST API for clinical interoperability. Patient, Observation, Condition, MedicationStatement. Used by EHR vendors and national health exchanges.
Biomedical ontologies
SNOMED CT, LOINC, RxNorm, HPO, GO, MONDO. Provide shared definitions of concepts — but each speaks a different serialisation format and identifier scheme.
Knowledge graphs
Wikidata, UniProt, ChEMBL, DisGeNET — public linked datasets exposing genes, drugs, diseases as SPARQL-queryable RDF. The connective tissue of the open biomedical web.
The common layer
RDF triples + shared URIs + SPARQL queries → federate across all four silos without moving data

This week we will build a working pipeline: CSV → OMOP CDM → R2RML mappings → Ontop virtual KG → SPARQL → FedBioMed federated analysis.

Day 2  ·  10:15 – 10:45

Introduction to RDF — three terms that connect everything

Every fact in the Semantic Web is expressed as a triple: Subject → Predicate → Object. An IRI (web address) names every node, so facts from any source can be joined without schema negotiations.

Subject
wd:Q1234
the patient / thing
Predicate
fhir:birthDate
the relationship
Object
"1985-03-12"
the value or linked node
IRIs as global names
http://www.wikidata.org/entity/Q42 unambiguously names Douglas Adams — in any dataset, in any language, without a lookup table. Prefixes (wd:, fhir:, schema:) are just shorthand.
Turtle syntax — the most readable RDF format
Triples end in .  ·  semicolon ; reuses the subject  ·  comma , reuses subject and predicate. That is almost all you need to read Turtle.
# A patient in Turtle
@prefix ex: <http://example.org/> .
@prefix fhir: <http://hl7.org/fhir/> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

ex:patient_001
  a fhir:Patient ;
  fhir:birthDate "1985-03-12"^^xsd:date ;
  fhir:gender "female" ;
  fhir:condition ex:condition_001 .

ex:condition_001
  a fhir:Condition ;
  fhir:code <http://snomed.info/id/38341003> .
# ^^ that SNOMED IRI links directly to the ontology

Key take-away: a triple is a sentence. "Patient 001 was born on 1985-03-12." Write enough sentences and you have a knowledge graph.

RDF hands-on — turning the mock dataset into triples

Using the HemaFAIR mock dataset (already loaded into OMOP CDM), we will express selected records as RDF triples in Turtle. Goal: see the same data in two representations and understand what the IRI layer adds.

1Open the Turtle editor (link on screen) — validates as you type

Day 2  ·  11:30 – 12:30

SPARQL — writing queries across OMOP and external knowledge graphs

Ontop exposes the OMOP CDM as a SPARQL endpoint. We can then add a SERVICE clause to pull in any public endpoint — Wikidata, UniProt, ChEMBL — and join the results in one query, without copying data.

# Find all patients with hypertension,
# then pull drug info from Wikidata

PREFIX fhir: <http://hl7.org/fhir/>
PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>

SELECT ?patient ?drug ?drugLabel
WHERE {
  # Local OMOP (via Ontop)
  ?patient a fhir:Patient ;
    fhir:condition ?cond .
  ?cond fhir:code
    <http://snomed.info/id/38341003> .
  ?patient fhir:medication ?drug .

  # Federate to Wikidata
  SERVICE <https://query.wikidata.org/sparql> {
    ?wdDrug wdt:P3345 ?drug ;
      rdfs:label ?drugLabel .
    FILTER(LANG(?drugLabel) = "en")
  }
}
What we will query together
Exercise 1 — List all patients with their SNOMED condition codes

Exercise 2 — Count patients per condition, ordered by frequency

Exercise 3 — Federate to Wikidata: for each drug in the mock dataset, retrieve its MeSH descriptor and ATC classification
Endpoint URLs for today
Local Ontop:
http://[server]/sparql

Wikidata:
https://query.wikidata.org/sparql

UniProt:
https://sparql.uniprot.org/sparql
Remember: SERVICE is live federation — the remote endpoint answers in real time. If Wikidata is slow, we switch to the cached mirror.

Day 3  ·  15:00 – 16:00

SPARQL hands-on — extracting features for federated learning

FedBioMed expects a structured feature table: one row per patient, columns for each clinical variable. We will write the SPARQL query that pulls exactly those columns from the Ontop endpoint — no ETL script, no data export.

Features FedBioMed needs (mock analysis)
Column OMOP source
patient_idperson.person_id
ageperson.birth_datetime
hypertensioncondition_occurrence (SNOMED 38341003)
on_aceidrug_exposure (RxNorm ACE inhibitors)
sbp_lastmeasurement (LOINC 8480-6)
1Identify the FHIR/RDF predicate for each feature in the Ontop mapping
2Write a SPARQL SELECT with one variable per feature column
3Use OPTIONAL for features that may not exist for every patient
4Export as CSV and load into the FedBioMed notebook
# Starter query — extend during the exercise

PREFIX fhir: <http://hl7.org/fhir/>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>

SELECT ?patient ?age
       ?hypertension ?on_acei ?sbp
WHERE {
  ?patient a fhir:Patient ;
    fhir:birthDate ?dob .
  BIND(year(now()) - year(?dob) AS ?age)

  # hypertension flag
  OPTIONAL {
    ?patient fhir:condition ?c .
    ?c fhir:code
      <http://snomed.info/id/38341003> .
    BIND(true AS ?hypertension)
  }
  # TODO: add ACE inhibitor and SBP blocks
}

→
③ Ontop virtual KG

→
⑤ FedBioMed

What changes at each site in the hackathon: only the DuckDB database (different patients). The R2RML mappings, Ontop config, and SPARQL query travel unchanged — that is the promise of standards-based interoperability.

Speaker notes — Introduction to RDF (30 min)

Start with the triple demo panel on the slide — ask participants to read it as a sentence: "Patient 001 was born on 1985-03-12." Emphasise that this is not a new idea; it is the way we already talk about data.

Work through the Turtle code block live. Show how semicolons let you write multiple facts about the same subject without repeating the URI. Show how the SNOMED IRI in the last triple is itself dereferenceable — open it in a browser.

Key concepts to land:
• IRI = global name, not just a web address
• Prefix = shorthand, nothing more
• Turtle = the most human-readable RDF format
• a is just shorthand for rdf:type

Common question: "Why not just use JSON?" — Answer: JSON has no globally shared property names. fhir:birthDate is the same property everywhere in the world; "birthDate" in JSON means whatever the author decided.

Timing: 5 min triple intro → 10 min live Turtle coding → 10 min questions → 5 min preview of hands-on.

Speaker notes — RDF Hands-on (30 min)

Participants write Turtle from the mock OMOP row shown on the slide. Use a shared live Turtle validator (shex.io works well) — paste results into a shared HackMD or Google Doc so the group can compare.

The deliberate trap in step 3 is the SNOMED IRI prefix — some participants will write https://snomed.info/id/ instead of http://snomed.info/id/. Let them discover this when they try to load both files into the same graph and a JOIN returns nothing.

Debrief questions:
• Did everyone end up with the same IRI for the same patient? (No — and that is a problem.)
• What would need to be agreed to make them the same? (A shared base IRI for the dataset.)
• What happened when you compared your SNOMED prefix with your neighbour's?

Timing: 5 min setup → 15 min individual/pair exercise → 10 min debrief.

Speaker notes — SPARQL combining data (60 min)

This session is the payoff for the morning. Participants now have Ontop running (Daphne's instance from Day 2 morning setup) and can query the OMOP data they loaded on Day 1 — as if it were always RDF.

Walk through the three exercises in order. Exercise 1 builds confidence. Exercise 2 introduces aggregation. Exercise 3 is where it gets interesting: SERVICE pulls from Wikidata without any data movement.

If Wikidata is slow: switch to the cached Wikidata mirror at https://qlever.cs.uni-freiburg.de/wikidata/sparql — same data, sub-second responses.

Common mistake: wrong Wikidata property IDs. Have the Wikidata item for Lisinopril open in a browser tab so participants can look up P3345 (RxNorm ID) themselves.

Timing: 10 min live demo of Exercise 1 → 10 min participants try Exercise 2 → 5 min debrief → 20 min Exercise 3 (pairs) → 15 min debrief + preview of Daphne's afternoon Ontop session.

Speaker notes — SPARQL for FedBioMed hands-on (60 min)

The starter query on the slide returns patient age and hypertension flag. Participants extend it to add two more columns: ACE-inhibitor exposure (drug_exposure filtered by RxNorm class) and last systolic blood pressure measurement (LOINC 8480-6).

Once the full query returns correct results, participants export CSV from the Ontop SPARQL UI (or via Python: pd.read_csv(endpoint_url + "?format=csv&query=" + urllib.parse.quote(q))) and hand it to the FedBioMed notebook.

Anticipated difficulty: OPTIONAL vs required — if a patient has no measurement, they should still appear with ?sbp = UNDEF rather than being dropped. Walk through this explicitly.

Connection to Day 4: Francesco's hackathon teams will use this exact CSV export pattern at multiple sites. The SPARQL query is the "data contract" between the KG layer and the FL layer.

Timing: 5 min recap + goal setting → 25 min hands-on (extend starter query) → 15 min export + FedBioMed load test → 15 min debrief.
